# java-bridge

# 📌 요구사항 요약
랜덤으로 위/아래 하나만 건널 수 있는 다리를 만들고,
사용자가 건널 수 있는 다리를 모두 맞히면 게임 종료
<br/>

#### 플로우
1. 게임 시작 문구를 보여줌
2. 사용자에게 3이상 20이하의 숫자를 다리 길이로 입력받음
3. 길이만큼 랜덤을 이용해 위/아래를 결정해 다리를 구성함
4. 라운드마다 플레이어로부터 U(위), D(아래)를 입력받음
5. 현재 진행상황에 맞게 다리 상태를 출력 (맞으면 O, 틀리면 X)
6. 게임을 실패하면 R(재시작), Q(종료)를 입력받고 결정함
7. 게임을 끝나면(성공/실패 모두) `최종 게임 결과(다리 상태)`와 `게임 성공 여부`, `총 시도한 횟수`를 출력함

## ⭐ 특이 사항
- [ ] 랜덤으로 0이 나오면 위(U), 1이 나오면 아래(D)
- [ ] 재시작해도 처음에 만든 다리를 활용함
  <br/>

# 📝 구현할 기능 목록
## 0. 모든 건 BridgeGame 위에서
- [ ] 제공된 `BridgeGame` 클래스를 활용할 것
- [ ] `BridgeGame`에서 `InputView` 및 `OutputView` 사용 금지
## 1. 게임 세팅
### a. 게임 시작 문구 출력
- [x] 제공된 `OutputView`로 시작 문구 출력
### b. 다리 길이 입력 받기
- 조건
- [x] 다리 길이 입력 문구 출력
- [x] 제공된 `InputView` 클래스를 활용
- [x] 사용자가 입력하는 값은 `camp.nextstep.edu.missionutils.Console`의 `readLine()`을 활용
- [x] 3 이상 20 이하의 숫자만 가능
- [x] 잘못된 값 입력 시, 정해진 예외 처리 후 다시 입력 받기
- [x] 길이 입력받고 빈 줄 출력
- 예외
- [x] 숫자 외 문자가 들어오는 경우
- [x] 3 이상 20 이하의 숫자가 아닌 경우 (음수, 0 포함)
### c. 다리 생성
- 조건
- [ ] 제공된 `BridgeMaker` 클래스를 활용해 구현 (필드 및 메서드 변경 불가)
- [ ] Random 값 추출은 제공된 `bridge.domain.BridgeRandomNumberGenerator`의 `generate()`를 활용
## 2. 게임 진행
- [ ] 각 라운드마다 증가되는 카운트 `BridgeGame`에 필드로 추가 (default: 1)
### a. 이동할 칸 입력 받기
- 조건
- [ ] `U`(위 칸)와 `D`(아래 칸) 중 하나의 문자만 입력 가능
- 예외
- [ ] 올바른 값이 아니면 예외 처리 후 다시 입력 받기
### b. 이동 적용 후, 진행 상황 출력
- 조건
- [ ] 이동할 수 있는 칸을 선택한 경우 `O` 표시
- [ ] 이동할 수 없는 칸을 선택한 경우 `X` 표시
- [ ] 선택하지 않은 칸은 ` ` 공백 한 칸으로 표시
- [ ] 다리의 시작은 `[`, 다리의 끝은 `]`으로 표시
- [ ] 다리 칸의 구분은 `|`(앞뒤 공백 포함) 문자열로 구분
- [ ] 현재까지 건넌 다리를 모두 출력
- [ ] 마지막에 빈 줄 출력
### c. 실패 시, 재시작/종료 여부 입력 받기
- 조건
- [ ] `R`(재시작)과 `Q`(종료) 중 하나의 문자를 입력 가능
- [ ] 재시작할 경우, 기존 진행 상황 초기화
- [ ] 재시작할 경우, 시도 횟수 증가
- 예외
- [ ] 올바른 값이 아니면 예외 처리 후 다시 입력 받기
## 3. 게임 종료
### a. 최종 게임 결과 출력
- [ ] 마지막 진행 상황을 그대로 출력
### b. 게임 성공 여부 출력
- [ ] 마지막 게임 결과에 따라 `성공`, `실패` 출력
### c. 총 시도한 횟수 출력
- [ ] 첫 시도부터 게임을 종료할 때까지 시도한 횟수 출력
  <br/>

# ⚠️ 프로그래밍 요구사항
- [ ] 프로그램 구현이 완료되면 ApplicationTest의 모든 테스트가 성공해야 한다.
- [ ] 함수(또는 메서드)의 길이가 10라인을 넘어가지 않도록 구현한다.
- [ ] indent(인덴트, 들여쓰기) depth를 3이 넘지 않도록 구현한다. 2까지만 허용.
- [ ] 3항 연산자를 쓰지 않는다.
- [ ] else 예약어 및 switch/case를 쓰지 않는다.
- [ ] 도메인 로직에 단위 테스트를 구현해야 한다.
- [ ] JUnit 5와 AssertJ를 이용하여 본인이 정리한 기능 목록이 정상 동작함을 테스트 코드로 확인한다.
- [ ] 메서드의 파라미터 개수는 최대 3개까지만 허용한다.
- [ ] InputView 클래스에서만 camp.nextstep.edu.missionutils.Console 의 readLine() 메서드를 이용해 사용자의 입력을 받을 수 있다.
- [ ] BridgeGame 클래스에서 InputView, OutputView 를 사용하지 않는다.
- [ ] 예외 상황 시 에러 문구를 출력해야 한다. 단, 에러 문구는 "[ERROR]"로 시작해야 한다.
- [ ] 사용자가 잘못된 값을 입력할 경우 IllegalArgumentException를 발생시키고, "[ERROR]"로 시작하는 에러 메시지를 출력 후 그 부분부터 입력을 다시 받는다.
